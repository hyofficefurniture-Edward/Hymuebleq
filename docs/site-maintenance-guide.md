# Hymueble 独立站长期运营维护说明书

适用范围：Hymueble / 鸿业家具西语独立站的日常内容发布、样式维护、本地验证、GitHub 发布和上线前检查。

本说明书面向长期运营：后续会持续新增产品、项目案例、Blog 文章和资源页内容。原则是让网站可以长期加内容，但不要让代码和样式再次堆成难维护的一团。

## 1. 目录分工

- `src/pages/`：页面路由。新增页面、行业页、项目页通常从这里开始检查。
- `src/components/`：可复用页面模块，例如 Header、Footer、表单、弹窗、卡片、展示模块。
- `src/data/`：站点数据、行业分类、导航、品牌信息、案例和图册配置。
- `src/styles/global.css`：全站样式入口，只负责按顺序导入 CSS partial。
- `src/styles/partials/`：拆分后的真实样式文件。
- `public/`：公开静态资源，例如图片、PDF、favicon。
- `docs/project-brain/`：项目策略、页面结构、验收记录和开发记录。
- `scripts/`：本地 QA、目录检查、发布前检查脚本。

## 2. CSS 维护规则

- `11-solutions-process.css`：解决方案区、路线桥接、标签和流程步骤卡。

- `14-resources.css`：资源页、Blog/Video 资源布局、决策卡、话题卡和资源 Showroom 卡。
- `20-showroom-quality.css`：Showroom 入口、Showroom 页面、质量流程相关样式。
- `30-contact-form-cta.css`：联系页、项目表单、表单状态和 CTA 区。

- `41-home-layout-responsive.css`：首页布局细节、分类卡片、移动端导航和字体比例覆盖。
- `42-trust-evidence.css`：信任证明、合作品牌、资质证据、证书标签和移动端流程滚动。
- `43-modal-project-summary.css`：Showroom 弹窗、项目解决方案摘要和项目行业入口。
- `44-catalog-factory-mobile.css`：酒店图册卡片、移动端项目卡片、质量流程和工厂/资质移动端布局。
- `45-final-type-legal-safeguards.css`：最终字体比例、移动端指标卡、法务页和最终移动端兜底。

### 修改原则

1. 不要把所有新样式继续塞回 `global.css`。它现在是入口清单，不是样式垃圾桶。
2. 新增一个可复用模块时，优先写组件，并把样式放到最接近它所属区域的 partial。
3. 新增产品、项目、Blog 内容时，优先复用现有模板和组件，不要为每篇内容写一套独立样式。
4. 不要随手改 `.btn`、`h1`、`.section-head`、Header、Footer 这类全站级选择器。必须改时，先确认会影响哪些页面。
5. 不要一边拆 CSS 一边改 class 名。重命名必须单独做，并配合全站搜索和浏览器验证。
6. 删除 CSS 前，必须先确认选择器没有被页面、组件、脚本、弹窗或移动端状态使用。

## 3. 发布新产品内容

新增产品内容时，先确认它属于哪一个业务入口。当前产品展示口径是 B2B 项目型图册，不是电商 SKU 货架：

- 酒店家具
- 办公家具
- 医疗家具
- 教育家具
- 住宅 / 公寓家具

建议流程：

1. 先按拉美西语搜索和采购习惯确认分类，不预设小类数量或每页产品数量。
2. 优先按空间、用途和项目场景归类，再用产品名称作为页内筛选标签。
3. 准备产品名称、西语标题、适用场景、材质、尺寸或可定制范围。
4. 优先从公司独立站站群和本地图册中筛选授权图片，但不得在前台重新增加站群导流入口。
5. 正式图片必须下载到本地、裁切、转 WebP，并确认图片文件名可读。
6. 填写 SEO 标题、描述、关键词和图片 alt。
7. 关联到对应行业页或图册入口。
8. 检查 WhatsApp / 询盘 CTA 是否清晰。

内容方向不要只写“便宜”和“好看”。B2B 买家更关心：项目适配、交期、质量控制、材料、包装、定制能力、从中国直采的沟通效率。

详细规则见 `docs/project-brain/04-素材/08-图册分类数量与素材来源规则.md`。

## 4. 发布项目案例

项目案例不是单纯展示图片，要服务于信任和询盘。

每个案例建议至少包含：

- 项目类型：酒店、办公室、学校、医疗、住宅等。
- 使用空间：客房、大堂、餐厅、会议区、病房、教室等。
- 供应内容：固定家具、活动家具、定制柜体、软装配套等。
- 采购价值：统一风格、批量交付、质量控制、成本效率、安装配合。
- 图片证据：实拍、工厂、包装、现场或图册页。
- 转化动作：查看图册、发送参考图、WhatsApp 沟通项目。

不要为了 SEO 大量复制相似案例文案。相似内容过多会降低页面质量，也会让维护变乱。

## 5. 发布 Blog 文章

Blog 文章的目标是获取搜索流量，并把读者引导到采购咨询。

建议文章类型：

- 采购指南：如何选择酒店家具供应商、如何评估中国家具工厂。
- 材料解释：板材、金属、软包、饰面、环保标准。
- 项目流程：从图纸、报价、打样、生产到装柜交付。
- 成本拆解：影响酒店家具报价的因素。
- 行业场景：酒店、公寓、办公室、学校、诊所的家具配置建议。

每篇文章发布前检查：

- 标题是否包含明确搜索意图。
- URL 是否短、清楚、可长期使用。
- 首段是否直接回答买家的核心问题。
- 是否有 2-4 个内部链接指向相关行业页、项目页或联系页。
- 图片是否有 alt 文案。
- 结尾是否引导“发送参考图 / 项目清单 / 图纸”。

## 6. 本地验证流程

每次改代码后，至少运行：

```bash
pnpm qa
```

内容、图册、路由或上线配置变更后，继续运行：

```bash
pnpm qa:prelaunch
```

涉及页面结构、Header、Footer、弹窗、移动端菜单、CSS 的改动，还要运行或人工检查：

```bash
pnpm qa:browser
```

需要判断 CSS 是否还有可清理项时，先运行：

```bash
pnpm qa:css:audit
```

这个命令只生成分析报告，不代表可以直接删除样式。删除前仍要确认选择器确实没有被模板、脚本、弹窗状态或响应式场景使用。

CSS 审计结果这样理解：

- `sourceUnreferencedClassCount`：优先看这个。它表示 CSS 里存在、但源码里没有直接搜到的 class。这个数字为 0 时，不建议为了减行数硬删。
- `duplicateSelectorCount`：重复 selector 不一定是坏事。很多重复来自桌面端、平板端、手机端、最后兜底覆盖，不能只看重复就合并。
- `topDuplicateSelectors`：优先检查这里列出的 selector。每次只处理一个小范围，例如只处理 Header，或只处理 Footer，不要跨区域大合并。
- `importantCount`：当前如果只出现在无障碍/减少动画这类基础保护里，可以接受；不要在新业务样式里继续增加 `!important`。

CSS 清理顺序建议：

1. 先处理文件归属和命名，让样式更容易找到。
2. 再处理明显无引用 class。
3. 最后才处理重复 selector，而且每次只处理一个区域。
4. 每次合并后必须跑 `pnpm qa`；涉及 Header、Footer、弹窗、移动端菜单时，再跑 `pnpm qa:browser`。

人工浏览至少覆盖：

- 首页
- 一个行业页
- 一个图册页
- 一个项目页
- Blog / Recursos 入口
- 联系页
- 手机端菜单
- WhatsApp 浮动按钮
- 图册弹窗和 Showroom 弹窗

## 7. GitHub 发布流程

建议每次上线按这个顺序：

1. 确认当前分支，不直接在不清楚状态下改主分支。
2. 先运行本地构建和 QA。
3. 用 `git status` 查看改动范围。
4. 只提交和本次目标相关的文件。
5. 提交信息写清楚目的，例如 `Split global CSS entrypoint and add maintenance guide`。
6. 推送到 GitHub。
7. 等待 GitHub Actions 的 CI 通过。
8. 在部署平台查看构建结果。
9. 打开线上地址做最后人工检查。

不要提交真实密钥、账号密码、Cookie、生产环境变量或客户敏感资料。示例配置只能使用占位符，例如 `YOUR_API_KEY_HERE`。

当前 CI 会检查：

- CSS 拆分结构没有被破坏。
- Astro 构建通过。
- 生成后的站内链接、本地资源、sitemap、表单和图册触发器通过静态检查。
- 图册来源映射和 lead-capture 规则通过检查。

当前 CI 运行的是 `pnpm qa`。它不替代上线前浏览器 QA，也不替代真实环境变量检查。上线前仍需本地或部署环境运行 `pnpm qa:browser` 和带真实配置的 `pnpm qa:prelaunch`。

## 8. 上线前验收清单

- 页面无明显中文残留，除非是内部文档。
- 所有主要 CTA 能打开正确动作。
- 移动端没有横向滚动。
- Header 菜单在手机端可打开、可关闭、无遮挡。
- Footer 链接和联系方式正确。
- 关键图片能显示，图片比例不严重变形。
- 主要页面有 title、description、canonical。
- sitemap、robots、favicon 正常。
- 构建命令通过。
- QA 脚本通过或明确记录未通过原因。

## 9. 后续治理节奏

CSS 后续不要一次性大重构。建议分三步：

1. 结构拆分：保持视觉不变，把样式从单文件拆到 partial。
2. 清理废弃样式：只删除确认 0 命中的旧样式块。
3. 组件化治理：把产品、项目、Blog 的重复页面结构沉淀成稳定组件和数据模板。

长期目标不是“CSS 文件越多越好”，而是每个文件有明确责任，新内容能按模板发布，修改一个模块时不会误伤全站。
