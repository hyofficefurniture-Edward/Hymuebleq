# Top 10 酒店家具厂家文章排版设计

## 目标

在不改动文章内容和既有 SEO 成果的前提下，将
`/recursos/top-10-fabricantes-muebles-hoteles-china-2026/`
改为已经确认的 A「精致采购期刊」视觉方向，并将方法论区块改成紧凑、接近老板原稿的表格形式。

最终页面应更像 Hymueble 的 B2B 采购期刊，而不是一篇直接套用全站默认样式的长文。桌面端强调受控阅读宽度、清晰层级和宽表突破；手机端保证正文舒适、方法论紧凑、宽表可滑动。

## 已确认的设计选择

采用 A 方案，但以“页面内的手术式实现”为准：

- 保留白色、暖米色、深海军蓝和品牌金色的现有品牌语言。
- 保留全部 10 家厂家及当前排名顺序。
- 仅突出 HYMUEBLE / Hongye Furniture Group，不把其他九家做成厚重卡片。
- 方法论桌面端保留三列表格，手机端使用同一张语义表转为紧凑纵向行。
- 比较矩阵和预算表在桌面端获得足够宽度，在手机端只在表格容器内横向滚动。
- 不移植汇报 demo 中的新增目录、锚点导航、标签文案、摘要文案或折叠交互。

另外两种方案不采用：

1. 只调整字号与全局间距：改动最小，但无法解决厂家长块、图片组和四张表格的层级问题。
2. 抽成新的共享文章组件或修改全站资源页 CSS：复用性更高，但会扩大影响范围，存在误伤其他页面的风险。

## 修改范围

业务页面只允许修改：

- `src/pages/recursos/top-10-fabricantes-muebles-hoteles-china-2026.astro`

文档只允许新增：

- 本设计文档。
- 后续对应的实施计划。

不修改：

- `src/layouts/BaseLayout.astro`
- `src/components/CTA.astro`
- `src/styles/` 下的全站样式
- Sitemap、导航、共享组件、其他路由或数据文件

## SEO 与内容冻结层

以下内容必须逐字、逐值保持不变：

- `pageTitle`
- `pageDescription`
- `keywords`
- `canonicalPath`
- `pageImage` 与绝对图片 URL
- `datePublished`
- `dateModified`
- `publicationDateLabel`
- Article JSON-LD
- FAQPage JSON-LD
- `BaseLayout` 的全部 SEO 参数
- H1 数量、文本和现有大小写
- H1–H4 的标签层级、文本及顺序
- 所有正文、列表、表格单元格、FAQ、CTA 和品牌签名文案
- 全部链接的目标与锚文本
- 7 张文章图片的顺序及 `src`、`alt`、`width`、`height`、加载属性
- 四张表格的内容和顺序

已有但不一致的地方也不在本次修正：

- 页面 Title 与 H1 的大小写差异。
- 可见 FAQ 与 FAQ Schema 的措辞差异。
- 文章中的公司事实、数据、排名和营销表述。

## 页面结构

### 页面作用域

在两段 JSON-LD 后增加一个仅用于样式隔离的 `.hotel-guide` 外层。页面专属 CSS 全部放在同一 Astro 文件底部，并以 `.hotel-guide` 为作用域。

不向共享 CSS 增加选择器，不改变 CTA 组件内部结构。

### Hero

保留当前 H1、两段引言和首图。Hero 使用左右双栏：

- 左侧为日期、H1 和原有两段引言。
- 右侧保留原图和图注。
- 桌面端使用接近 45/55 的比例，手机端回到单栏。
- 首图保持当前源文件和属性，仅调整容器裁切、圆角和阴影。

### 正文阅读节奏

普通正文区使用约 800–860px 的阅读宽度；比较矩阵等宽内容使用约 1160–1280px。统一：

- H2 与首段之间的距离。
- 段落、列表和图片之间的垂直节奏。
- 正文字号与约 1.7 的行高。
- 区块上下留白，桌面端约 64–88px，手机端约 46–56px。

不新增章节名称、编号或目录文字。

### 方法论

继续使用现有 `<table>`，不复制一套移动端内容：

- 桌面端：深色表头、细分隔线、浅斑马纹、紧凑单元格。
- 手机端：隐藏视觉表头，将每个现有 `tr` 转为紧凑纵向行。
- 手机端通过 CSS 和单元格属性显示“Evalúa / Importa”辅助标签；表格正文不改。
- 目标是比六张大卡片明显更矮，并保留老板原来的表格感。

### Top 10 厂家

将每一家厂家现有节点原样包入独立的 `.manufacturer-entry`：

- 01–10 全部显示，顺序不变。
- 普通厂家采用连续分隔线式排版，不做十张厚卡。
- 厂家标题中的现有序号继续使用，不另外生成一套序号。
- 长标题允许安全换行，手机端不得溢出。

HYMUEBLE 使用 `.manufacturer-entry--featured`：

- 浅米色背景、细金色边框和克制的内边距。
- 保持整个鸿业长内容自然高度，不设置固定高度。
- 现有工厂图、安装内容、案例、产能表和最后的 lobby 图全部留在同一个厂家块中。
- 三张连续案例图外包为 `.project-gallery`，桌面三列、平板两列、手机一列；不改图片本身的任何属性。

### 表格

- 产能表使用紧凑两列样式。
- 比较矩阵使用宽内容容器，并通过 CSS 轻度突出第二行 HYMUEBLE。
- 预算表使用同一视觉系统。
- 手机端比较矩阵和预算表设置合理的最小宽度，在自身 wrapper 内横向滚动。
- 页面本身不得产生横向滚动。

### 常见错误、FAQ 与结尾

- 常见错误继续使用原有 `<ol>`，桌面三列、手机单列；不拆写或压缩文字。
- FAQ 将每组现有 `h3 + p` 仅包入 `.faq-item`，答案始终可见；不改为折叠 `<details>`。
- 现有 CTA 组件保持原样。
- 品牌签名只做间距、细分隔线和字色处理。

## 响应式与可访问性

- 验收视口：1440×900、1024×768、390×844。
- 390px 下不允许标题、正文、图片、FAQ 或厂家名称溢出。
- 仅宽表容器允许横向滚动。
- 表格继续保留真实表格语义。
- FAQ 不隐藏答案。
- 不用 CSS 隐藏正文或 SEO 文本。
- 保留现有焦点、链接和 CTA 行为。

## 验收方法

### 生成页面零漂移指纹

修改前和修改后分别从重新构建的页面导出 JSON 指纹，对比必须为空。指纹包括：

- Title、Meta description、keywords、robots、canonical。
- Open Graph 与 Twitter 的 title、description、URL、image。
- H1 数量和精确文本。
- H1–H4 的层级、文本与顺序。
- Article 与 FAQPage JSON-LD 的解析结果；FAQ 必须为 6 项。
- 可见发布日期和两项 Schema 日期。
- 去除 `script` 与 `style` 后、规范化空白的完整正文 `textContent`。
- 四张表格逐行逐格内容，数据行分别为 6、6、10、4。
- 7 张文章图片按顺序的关键属性。
- 主内容链接的目标与锚文本。
- Sitemap 中该 URL 精确出现一次。

### 项目 QA

实施后运行：

- `pnpm qa:css`
- `pnpm build`
- `pnpm qa:static`
- 本地服务上的 `pnpm qa:browser`
- `git diff --check`

已知的图册 PDF 本机绝对路径 CI 问题不属于本次范围，若再次出现必须单独报告，不能通过修改文章或降低本次验收标准绕过。

### 视觉 QA

逐段检查：

- Hero 在桌面和手机端均无异常大空白。
- 正文行宽和段落节奏一致。
- 方法论桌面紧凑，手机不挤成三列。
- 10 家厂家全部可见，鸿业重点明确但不破坏阅读。
- 鸿业图片组不畸变、不裁掉关键证据。
- 四张表格可读，手机只在表格内部滑动。
- 常见错误、FAQ、CTA 与品牌签名无重叠。
- WhatsApp 浮层不遮挡表格或主要文本。

## 风险控制

- Astro 页面专属 CSS 必须使用 `.hotel-guide` 作用域，避免影响其他路由。
- 不清理当前未使用的 `SectionHead` import；它是既有问题，不属于本次排版任务。
- 不提交既有未跟踪的 `.agents/` 或 `AGENTS.md`。
- 不顺手修正事实、SEO、拼写、链接、日期、图片或共享样式。
