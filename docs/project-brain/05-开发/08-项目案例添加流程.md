# 项目案例添加流程

适用场景：给 Hymueble 西语独立站新增酒店、办公、医疗、教育、住宅类目的项目案例。

本流程的目标不是“多写几张卡片”，而是让每个案例同时服务三件事：

- 类目页视觉更丰富。
- 买家能点进详情页，看清项目范围和证据。
- Google 能识别这是独立、可索引、主题明确的项目页面。

## 1. 先找来源，不先写文案

优先从集团已有站群找来源：

- 酒店：`https://www.hyhotelfurniture.com/projects.html`
- 办公：`https://www.hyofficefurniture.com/business-furniture-projects.html`
- 医疗：`https://www.hyhealthcarefurniture.com/cases.html`
- 教育：`https://hyeducationfurniture.com/`
- 住宅真实公寓案例：优先查 `https://www.hyhotelfurniture.com/` 的 apartment case 页面和本地 `PDF成功案例 2024-2025/公寓/`
- 住宅解决方案场景：`https://fenmicasa.com/`

摘取时只拿可以确认的信息：

- 项目名
- 类目
- 地区或国家
- 空间范围
- 供应内容
- 可证明的亮点
- 来源 URL

不确定的年份、房间数、金额、客户关系，不要编。

### 1.1 先做证据分级

正式网站只把 A 级项目叫真实案例。

| 等级 | 前台西语标签 | 使用口径 |
|---|---|---|
| A | `Caso documentado` | 有同名 PDF、图片、视频、安装图或内部案例包，且图片与项目匹配，可以作为真实项目案例。 |
| B | `Referencia publicada` | 有集团站群、知识库或 PDF 线索，但当前图片/细节不足，只作为项目参考。 |
| C | `Escenario de solución` | 来自产品、服务、风格或采购场景，不是真实同名项目，只作为解决方案场景。 |

如果一个类目不够 6 个 A 级真实案例，可以用 B/C 补足视觉和 SEO 覆盖，但前台必须明确成“参考/场景”，不要写成真实项目。

## 2. 图片先用本地素材

项目卡片和详情页图片先放在 `public/assets/hymueble/`。

当前可直接复用的目录：

- 酒店案例图：`public/assets/hymueble/casos/`
- 办公图：`public/assets/hymueble/oficinas/`
- 医疗图：`public/assets/hymueble/salud/`
- 教育图：`public/assets/hymueble/educacion/`
- 住宅图：`public/assets/hymueble/residencial/`

如果从站群下载真实案例图，先重命名为可读英文或拼音，不使用 `1.jpg`、`微信图片...` 这类文件名。

如果从 PDF 截图或裁图，先自己检查预览：

- 太糊、太窄、构图很丑的图，不放首页或类目项目卡片。
- 只有包装、边角、水印、局部材料的图，不放强案例封面；最多放详情页证据模块。
- 同项目图片不足时，宁愿降级为 `Referencia publicada`，不要用别的项目图冒充。

### 2.1 PDF 裁图的实际流程

本次医疗案例批次使用的流程可以复用：

1. 用 `pdftoppm` 把 PDF 渲染成 PNG，临时文件放 `tmp/pdfs/<batch>/`。
2. 先生成整页分段联系表，例如 `output/pdf/<batch>/health-pdf-segment-contact-sheet.jpg`，自己看一遍页面结构。
3. 只从真实空间照片里做候选裁图；不要直接截带大段正文、页脚、社媒按钮、黄色标题条的截图。
4. 生成候选联系表后再看一次，发现丑裁切就重裁。
5. 如果 PDF 裁图总是带标题、正文、页脚或社媒按钮，回到同名公开项目页下载原图；不要勉强把 PDF 残片放进正式卡片。
6. 最终 WebP 放到 `public/assets/hymueble/casos/`，命名用项目名 + 空间名，例如 `ganjiang-waiting-area.webp`。
7. `site.ts` 只引用最终精选图，不引用中间候选图。

这一步的判断标准很简单：如果截图像 PDF 残片，而不是一张能代表项目空间的照片，就不要放到正式卡片里。

## 3. 数据写在 `src/data/site.ts`

项目案例统一写进 `projects` 数组，不要为单个案例新建页面文件。

新增时复制这个结构：

```ts
{
  name: "Project Name",
  seoTitle: "Project Name, referencia de mobiliario ...",
  seoDescription: "150-160 caracteres左右，包含项目名、类目关键词和 B2B 采购意图。",
  seoKeywords: ["项目名", "类目关键词", "mobiliario ...", "proyecto ...", "cotización B2B"],
  sector: "Hotel",
  region: "País o región",
  scope: "一句话说明空间范围和供应内容。",
  facts: ["事实 1", "事实 2", "事实 3"],
  evidenceLabel: "Caso documentado",
  alt: "图片 alt，说明类目和项目名",
  image: img("casos/example.webp"),
  homeImage: img("casos/example-home.webp"),
  hubImage: img("casos/example-hub.webp"),
  listingImage: img("casos/example-listing.webp"),
  detailImage: img("casos/example-detail.webp"),
  href: "/proyectos/hoteles/project-slug/",
  sourceUrl: "https://source-site.example/project.html",
  details: {
    type: "项目类型",
    year: "不要乱写年份；不确定就写 Referencia publicada...",
    rooms: "详情页的空间范围",
    finishes: "材料、饰面、生产、交付或支持方式",
    evidence: ["证据点 1", "证据点 2", "证据点 3"],
    supportImage: img("casos/support.webp"),
    overview: "2-3 句项目背景，写可证实的信息，不编造。",
    spaces: ["可见空间 1", "可见空间 2", "可见空间 3"],
    procurementValue: "从采购经理角度说明这个案例能证明什么。",
  },
}
```

`overview` 是详情页可信度的核心字段。A 级真实案例必须人工写 `overview`、`spaces`、`procurementValue`；B/C 项目如果暂时缺少 `overview`，详情页模板会生成保守说明，但后续补证据时应优先改成人工项目解读。

## 4. SEO 字段怎么写

每个项目都必须有：

- `seoTitle`：包含项目名 + 类目关键词。
- `seoDescription`：用自然西语说明项目用途，建议 150-160 字符左右。
- `seoKeywords`：项目名、类目词、采购词、B2B 词。
- `alt`：不要只写项目名，要写“referencia de mobiliario...”。
- `href`：短、清楚、无重音符号，末尾保留 `/`。
- `evidenceLabel`：只有需要手动覆盖时才写；默认逻辑会按来源判断。住宅解决方案场景通常显示 `Escenario de solución`。

示例：

```ts
seoTitle: "Walmart Costa Rica, referencia de mobiliario corporativo",
seoDescription: "Walmart Costa Rica como referencia de mobiliario de oficina para revisar espacios corporativos regionales y compras B2B por proyecto.",
seoKeywords: ["Walmart Costa Rica", "mobiliario corporativo", "muebles de oficina", "oficinas Latinoamérica", "proyecto B2B"],
```

不要为了 SEO 堆砌重复词。页面要像给采购经理看的案例说明，不像关键词清单。

## 5. 路由会自动生成

只要 `href` 写成这种格式：

```ts
/proyectos/oficinas/walmart-costa-rica/
```

详情页会由这个模板自动生成：

```bash
src/pages/proyectos/[sector]/[project].astro
```

所以以后新增项目时，一般只改 `src/data/site.ts`。

## 6. CSS 应该怎么看

项目卡片样式主要在：

```bash
src/styles/partials/10-home-hero-cards.css
src/styles/partials/40-visual-overrides.css
src/styles/partials/44-catalog-factory-mobile.css
```

当前项目卡片复用：

- `.card-grid`
- `.project-card`
- `.project-facts`
- `.home-project-rail`

不要为单个项目写专属 CSS。

只有出现这些问题才考虑改 CSS：

- 6 张卡片在手机端横向溢出。
- 卡片文字把按钮挤坏。
- 图片比例明显不统一。
- 类目页项目区视觉密度过高。

如果要改，先改对应 partial，不要把新样式写进 `src/styles/global.css`。

## 7. 改完后必须检查

至少运行：

```bash
pnpm qa
```

新增项目后还要重点确认：

- 每个类目项目数是否为 6。
- 每个项目详情页是否生成。
- sitemap 是否包含新增项目 URL。
- 页面没有中文残留。
- 图片路径没有 404。
- 移动端没有横向滚动。

可以用本地预览抽查：

```bash
pnpm run dev
```

重点看：

- `/hoteles/`
- `/oficinas/`
- `/salud/`
- `/educacion/`
- `/residencial/`
- `/proyectos/hoteles/`
- 任意一个新增项目详情页

## 8. 本次实现口径

本次补齐为 5 个类目各 6 个项目，共 30 个项目。

主首页只展示精选项目，避免首页变成长案例墙；完整 6 个项目展示放在对应类目主页和类目项目页。

住宅类目当前没有足够公开真实项目页，因此采用 Fenmi Casa 住宅服务、产品和内容方向转写为“解决方案场景”。前台不叫真实项目。后续如果拿到真实别墅、公寓、开发商项目资料，可以直接替换这些参考型场景，并按 A/B/C 重新分级。
