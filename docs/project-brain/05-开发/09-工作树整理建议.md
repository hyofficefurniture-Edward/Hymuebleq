## 3. 暂不建议提交

### 临时产物

不建议提交：

- `tmp/`
- 临时原图下载文件
- PDF 渲染中间页
- 大量 contact sheet /候选截图，除非被明确选入验收证据

建议处理：

- 如果仍需保留给下一轮查证，先留在本地但不要 stage。
- 如果确认不再需要，后续可在用户批准后清理 `tmp/`。

### 需要确认归属的文件

这些文件可能有价值，但建议单独确认后再纳入：

- `.github/workflows/ci.yml`
- `docs/site-code-operation-sop.md`
- `docs/site-maintenance-guide.md`
- `.env.example`

原因：

- 它们属于工程治理/部署/维护层，不完全等同于本轮案例和页面模块修复。
- `.env.example` 必须确认只含占位值，不能包含真实联系方式、密钥或私有 endpoint。

## 4. 推荐下一步

1. 先运行 `git diff --stat` 和 `git status --short`，确认没有意外文件。
2. 先做 A 组 CSS 架构提交，再做 B 组案例证据提交，再做 C 组类目模块修复提交，最后做 D 组记录提交。
3. 每组提交前只 stage 对应文件，避免把 `tmp/` 和无关治理文件混进去。
4. 提交前至少跑一次 `pnpm qa`；如果改动包含视觉/交互模块，再跑 `pnpm qa:browser`。
5. 正式上线前仍需补真实 WhatsApp、业务邮箱、表单接收 endpoint，并跑 `pnpm qa:launch`。
