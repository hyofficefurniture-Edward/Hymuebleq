# 项目案例证据分级与 SEO 流程

- 状态：执行口径
- 最后更新：2026-07-02
- 适用范围：首页精选项目、5 个类目主页项目区、5 个类目项目页、项目详情页
- 核心原则：正式网站只把 A 级项目称为真实案例；不够 6 个时，用“项目参考 / 解决方案场景”补足，但不叫真实项目。

## 1. 证据分级

| 等级 | 前台西语标签 | 可否叫真实案例 | 判定标准 | SEO 写法 |
|---|---|---:|---|---|
| A | `Caso documentado` | 可以 | 有同名 PDF、图片、视频、安装图或内部案例包；图片与项目匹配；事实字段不靠猜 | 可以围绕项目名、行业词、空间范围写独立项目页 |
| B | `Referencia publicada` | 不建议 | 集团站群或知识库有同名/同类公开来源，但当前图可能是类目通用图，或细节不足 | 以 referencia / proyecto publicado / alcance comparable 为主，不写未证实年份、数量、金额 |
| C | `Escenario de solución` | 不可以 | 主要来自产品、服务、风格、采购场景或 Fenmi Casa 内容方向，不是同名真实项目 | 以 escenario / referencia visual / solución residencial 为主，服务长尾搜索和转化 |

## 3. 新增项目的操作流程

1. 先建证据卡，不先写西语文案：
   - 项目名、类目、国家/城市、空间范围、来源 URL、本地 PDF/图片/视频路径。
   - 缺失字段写“待补”，不要猜。
2. 判定 A/B/C：
   - 同名项目 + 同项目图片/视频/PDF + 可公开事实 = A。
   - 只有公开站群页或知识库稿，但图片不足 = B。
   - 只有产品/风格/采购场景 = C。
3. 写 `src/data/site.ts`：
   - A：可用 `Caso documentado`，可写项目名 + 行业关键词。
   - B：保留 `Referencia publicada`，避免 “real case / completed project” 这种过度口径。
   - C：写 `Escenario de solución`，标题和描述用 scenario/reference，不写真实客户。
4. 图片处理：
   - PDF 截图先生成预览，自己检查是否清晰、构图是否完整、是否有大水印/边角/丑裁切。
   - 很丑的截图不放首页和类目卡片；最多放详情页证据模块，或暂时不用。
   - 图片文件名用英文可读命名，不用 `1.jpg`、`微信图片...`。
5. SEO 字段：
   - `seoTitle`：项目名/场景名 + 类目关键词。
   - `seoDescription`：自然西语，150-160 字符左右，说明行业、空间、采购意图。
   - `seoKeywords`：项目名、行业词、家具词、B2B 采购词，不堆砌。
   - `alt`：描述图片用途和类目，不只写项目名。
6. CSS：
   - 项目卡片主样式在 `src/styles/partials/10-home-hero-cards.css`。
   - `.meta .evidence-badge` 是证据标签样式。
   - `.project-card`、`.project-facts`、`.card-grid` 是项目卡片结构，不为单个项目写专属 CSS。
