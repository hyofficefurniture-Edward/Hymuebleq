- 状态：进行中
- 最后更新：2026-06-29
- 负责人：主对话总控
- 关联资料：多线程工作法、项目大脑、验收记录
- 关联页面：全站
- 下一步：每轮大改后追加一段复盘，沉淀更好的协作方式。

## 本轮有效做法

1. 先把策划案、导航图、页面树写入项目大脑，再动代码。
2. 把“老板演示必须不 404”作为第一条工程红线。
3. 主对话负责源码实施，子对话只做资料、规划和 QA，避免并发修改冲突。
4. 每次开发后用 `pnpm run build`、内部链接检查、浏览器表单/弹窗/移动菜单检查闭环。
5. 所有关键结论回写 `docs/project-brain`，减少长对话压缩后丢失上下文的风险。

## 适合你以后给我的授权方式

可以用这种结构：

```text
北极星：严格以某文件/某资料为最高依据。
目标：X 小时内完成可预览版本。
边界：不接后端、不填真实敏感信息、不发布上线。
优先级：先补点击闭环，再做视觉，再做内容证据。
验收：build、链接、表单、弹窗、移动端、核心页面截图。
记忆：所有结论写入 docs/project-brain。
多线程：主对话写代码，子对话只读报告，避免冲突。
```

## 本轮自我优化点

- 不再只盯第一版五个页面，而是按页面树补齐所有会被点击到的 URL。
- 不把“轻量页”做成空占位，而是让它至少有真实图片、行业说明、目录/项目 CTA 和流程。
- 不把资料合规性反复拿出来确认，按用户授权直接使用，但仍避免写入真实敏感联系方式。
- 每次新增路由后立即跑构建和链接检查，而不是等最后一起排错。

## 下一轮应避免

- 不要让多个线程同时改 `src/`，否则容易互相覆盖。
- 不要把证书、案例、图册一股脑堆页面；先建证据库，再选择最有转化价值的材料。
- 不要只追求页面数量；老板演示最重要的是路径流畅、信任清楚、CTA 清楚、无明显错位。
