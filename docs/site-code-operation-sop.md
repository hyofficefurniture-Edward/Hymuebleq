# Hymueble 独立站代码操作 SOP

适用对象：不以写代码为主要工作，但需要长期运营 Hymueble / 鸿业家具西语独立站的人。

这份 SOP 的目的不是教你变成程序员，而是让你知道：

- 每次改网站前应该先看什么。
- Blog、产品、项目案例应该怎么长期增加，才不会把网站越改越乱。
- CSS 样式应该怎么维护，才不会回到一个 `global.css` 装所有东西的状态。
- 使用 WorkBuddy / AI 工具时，怎么下任务、怎么限制范围、怎么验收结果。
- 改完以后应该跑哪些检查，什么情况可以继续，什么情况必须停下来。

第一版范围：日常代码维护、内容发布、CSS 维护、QA 检查、GitHub 发布前准备、WorkBuddy 协作。域名、部署平台、真实表单服务配置不在本 SOP 深入展开，后续可以单独写上线部署 SOP。

## 0. 先读这一页

### 这份 SOP 解决什么问题

你的网站以后会不断增加 Blog、产品、项目案例、图片和资料。如果没有规则，网站很容易变成：

- 文字到处复制。
- 图片路径到处散落。
- 每个新页面都写一套自己的结构。
- 每次改一个卡片，都顺手改全站 CSS。
- 最后又出现类似以前 `global.css` 很大、很难维护的问题。

这份 SOP 要防止的，就是这种长期运营后的混乱。

### 三条红线

1. 不提交真实密钥。
   - 包括 API key、密码、Cookie、后台账号、数据库地址、真实生产 token。
   - 示例配置只能写 `YOUR_API_KEY_HERE` 这类假值。

2. 不乱删 CSS。
   - CSS 看起来重复，不代表没用。
   - 很多重复是桌面、平板、手机、弹窗、最终兜底造成的。

3. 不跳过 QA。
   - QA 就像体检。
   - 改完不检查，等到上线后发现问题，成本更高。

### 出问题时先停手

如果你发现：

- 页面突然乱了。
- 手机端出现横向滚动。
- 图片不显示。
- QA 命令失败。
- WorkBuddy 改了很多你没要求的文件。

先不要继续扩大修改范围。先记录现象，再看改动范围，再跑检查。

### 案例

错误做法：看到 `global.css` 曾经很大，就想继续删 CSS 行数。

正确做法：先跑 `pnpm qa:css:audit`，看有没有明确未引用 class；没有证据就不要删。

## 1. 网站文件地图

你可以把这个项目理解成一栋房子。

### 主要目录

- `src/pages/`
  - 页面入口。
  - 你可以理解为网站里的每个房间。
  - 例如首页、联系页、项目页、图册页、Blog 页。

- `src/components/`
  - 可复用模块。
  - 你可以理解为可以搬来搬去的家具。
  - 例如 Header、Footer、CTA、卡片、FAQ、表单、弹窗。

- `src/data/`
  - 网站内容数据。
  - 当前很多行业分类、项目案例、图册信息、FAQ、联系配置都集中在 `src/data/site.ts`。
  - 你可以理解为内容仓库。

- `src/styles/`
  - 样式文件。
  - 你可以理解为装修风格。
  - 当前 `src/styles/global.css` 只是入口清单，真实样式在 `src/styles/partials/`。

- `public/`
  - 公开素材。
  - 例如图片、PDF、favicon。

- `scripts/`
  - 检查工具。
  - 例如 CSS 检查、静态链接检查、图册检查、浏览器 QA。

- `docs/`
  - 项目说明、维护文档、验收记录、SOP。

### 找文件的默认规则

- 想改页面内容：先看 `src/pages/` 和 `src/data/`。
- 想改可复用模块：先看 `src/components/`。
- 想改样式：先看 `src/styles/partials/`，不要直接往 `global.css` 写。
- 想换图片：先确认图片在 `public/` 下的位置。
- 想看检查命令：先看 `package.json` 里的 `scripts`。

### 案例：改首页一段文字

先找首页文件：

```bash
src/pages/index.astro
```

如果文字来自数据，再看：

```bash
src/data/site.ts
```

不要一开始就改 CSS。文字问题通常不是样式问题。

### 案例：换一张图

先确认新图放在 `public/` 下的正确素材目录，再改页面或数据里的图片路径。不要把图片放到随机位置，也不要用看不懂的文件名，例如 `111.png`。

### 案例：改按钮颜色

不要直接搜索 `.btn` 后随便改。`.btn` 是全站按钮基础样式，可能影响首页、联系页、弹窗、Footer、移动端菜单。应该先判断是全站按钮都要改，还是只改某一个区域。

## 2. 每次动代码前的准备 SOP

### 适用场景

只要你准备改网站，不管是改文字、加产品、换图片、改 CSS、让 WorkBuddy 操作，都先走这一章。

### 操作步骤

1. 确认当前项目目录。
   - 当前项目目录是：

```bash
/Users/yoyo/Documents/鸿业家具西班牙语独立站搭建
```

2. 看当前改动状态。

```bash
git status --short --branch
```

白话解释：

- 这个命令会告诉你当前在哪个分支。
- 会列出哪些文件已经被改过。
- 如果里面有很多你不认识的文件，不要急着继续改。

3. 明确本次只做一类事情。

推荐拆分：

- 只改 Blog。
- 只加产品。
- 只加项目案例。
- 只改 CSS。
- 只做发布前检查。

不要把“改 CSS、加产品、改 Header、改表单、改 Blog”混在同一次任务里。

4. 确认是否涉及敏感信息。

看到这些内容时要小心：

- `.env`
- `.env.local`
- API key
- password
- token
- Cookie
- secret
- 数据库地址

真实密钥不能写进文档、截图、日志、提交信息、示例代码。

### 常见错误

- 没看 `git status` 就开始改。
- 不知道哪些文件已经被别人或之前的任务改过。
- 把真实账号密码写到文档里。
- 一次任务改太多范围，出问题后不知道是哪一处造成的。

### 检查方式

改之前和改之后都可以运行：

```bash
git status --short --branch
```

你要能回答：

- 这次改了哪些文件？
- 这些文件是否都和本次任务有关？
- 有没有看起来无关的改动？

### 案例

如果你只是新增一篇 Blog，但 `git status` 显示 CSS、Header、Footer、表单都被改了，这就是风险信号。应该先停下来，要求说明为什么这些文件会被改。

## 3. 内容、结构、样式三分法

### 为什么这一章很重要

以前 `global.css` 的问题，本质不是“文件名叫 global 所以不好”，而是所有样式都堆到一个地方，后期不知道谁影响谁。

以后 Blog、产品、项目案例也会遇到类似问题。不是所有东西都堆进 `global.css`，但可能变成：

- 每篇 Blog 都复制一套页面。
- 每个产品都新写一个卡片样式。
- 每个案例都复制整页代码。
- 内容、页面结构、CSS 混在一起。

这样长期看也会变乱。

### 三个概念

1. 内容
   - 写什么。
   - 例如标题、描述、产品参数、项目介绍、图片说明。

2. 结构
   - 内容放在哪里。
   - 例如 Hero、卡片、列表、详情页、FAQ、CTA。

3. 样式
   - 长什么样。
   - 例如颜色、间距、字体大小、卡片圆角、手机端排版。

### 默认原则

- 新增内容，优先改数据或内容文件。
- 多个内容需要同一种展示方式，才考虑新增组件。
- 只有现有样式无法支持长期复用场景，才考虑新增 CSS。
- 不要为单篇 Blog、单个产品、单个案例写专属全局 CSS。
- 不要把真实样式写回 `src/styles/global.css`。

### 决策顺序

每次要改网站，先问三个问题：

1. 我是在改内容吗？
   - 是的话，优先找 `src/data/` 或对应页面内容。

2. 我是在改结构吗？
   - 是的话，优先找 `src/pages/` 或 `src/components/`。

3. 我是在改样式吗？
   - 是的话，优先找 `src/styles/partials/`，并确认影响范围。

### 常见错误

- 为了写 Blog 去改全站按钮样式。
- 为了上传一个产品复制一个完整页面。
- 为了加一个案例新建一套 CSS。
- 为了局部文字换行，去改全站 `h1`。

### 案例

你要新增“酒店客房床头柜”。

正确路径：

1. 先判断这是产品内容。
2. 找现有产品或图册数据结构。
3. 复用现有产品卡片或目录卡片。
4. 配好图片、alt、分类、询盘入口。
5. 只有现有卡片无法长期承载这种产品类型时，才考虑新增组件或 CSS。

错误路径：

1. 复制一个页面。
2. 改一点文字。
3. 为这个产品写新的 CSS。
4. 最后产品越来越多，每个页面都不一样。

## 4. Blog 长期发布 SOP

### 适用场景

你准备写采购指南、材料说明、项目流程、成本拆解、行业场景文章时，使用这一章。

注意：你的目标不是单纯为了 SEO 堆关键词，而是让网站在清晰逻辑下长期稳定运行。SEO 是结果之一，不是破坏结构的理由。

### 当前状态

当前 Blog 更像一个固定资源页：

```bash
src/pages/recursos/blog.astro
```

未来如果要大量发布文章，应该优先建设统一的内容结构或文章模板，而不是每篇文章复制一份独立页面。

### 每篇 Blog 应该准备的字段

- 标题。
- URL slug。
- 摘要。
- 分类。
- 主图。
- 图片 alt。
- 正文。
- 相关内部链接。
- CTA。
- 发布时间或更新日期。
- 适合的目标读者。

### 操作步骤

1. 先写内容大纲。
   - 文章要解决什么采购问题？
   - 读者看完以后下一步应该做什么？

2. 确认文章类型。
   - 采购指南。
   - 材料解释。
   - 项目流程。
   - 成本拆解。
   - 行业场景。

3. 复用统一结构。
   - 不要每篇文章新建一种布局。
   - 先复用现有资源页/Blog 模板。

4. 准备图片。
   - 文件名要可读。
   - 图片 alt 要说明图片内容。
   - 不要用无意义文件名，例如 `1.jpg`。

5. 添加内部链接。
   - 指向相关行业页、图册页、项目页或联系页。
   - 不要让 Blog 成为孤岛。

6. 添加 CTA。
   - 例如发送项目清单、发送参考图、查看图册、联系项目顾问。

### 不要做什么

- 不要为了一篇 Blog 新增全局 CSS。
- 不要每篇 Blog 复制一整个 `.astro` 页面后改文字。
- 不要为了关键词堆很多相似文章。
- 不要让文章只有文字，没有下一步转化动作。

### 检查方式

改完后至少运行：

```bash
pnpm qa
```

如果改了页面布局或移动端展示，再运行：

```bash
pnpm qa:browser
```

### 案例：写一篇“如何选择酒店家具供应商”

正确做法：

1. 标题围绕真实采购问题。
2. 正文按采购逻辑写：工厂能力、材料、打样、交期、包装、出口经验。
3. 内部链接指向酒店家具页、酒店图册页、项目案例页、联系页。
4. 复用 Blog 模板。
5. 不新增专属 CSS。

错误做法：

1. 为这篇文章新建一个完全不同的页面。
2. 为这篇文章写专属按钮样式。
3. 只堆关键词，不引导采购下一步。

## 5. 产品上传 SOP

### 适用场景

你要增加酒店家具、办公家具、医疗家具、教育家具、住宅家具等产品内容时，使用这一章。

### 产品内容默认字段

每个产品至少准备：

- 产品名称。
- 所属行业或分类。
- 应用场景。
- 简短描述。
- 主图。
- 图片 alt。
- 可定制点。
- 材料或工艺要点。
- 适合项目类型。
- 询盘入口。

### 操作步骤

1. 判断产品属于哪个入口。
   - 酒店家具。
   - 办公家具。
   - 医疗家具。
   - 教育家具。
   - 住宅 / 公寓家具。

2. 查找现有数据结构。
   - 当前大量站点数据集中在：

```bash
src/data/site.ts
```

3. 查找现有卡片和目录模板。
   - 不要先新建页面。
   - 先确认有没有现成产品卡片、图册卡片、分类卡片可复用。

4. 准备图片和 alt。
   - 图片放到合适的 `public/` 目录。
   - alt 要说明图片内容，而不是堆关键词。

5. 关联询盘入口。
   - 产品最终要能引导到联系、图册、WhatsApp 或询盘动作。

### 不要做什么

- 不要为单个产品新建一套 CSS。
- 不要为单个产品复制完整页面。
- 不要把产品图片散放到无关目录。
- 不要只写“高端、便宜、质量好”这种空泛描述。

### 检查方式

内容或图册变更后运行：

```bash
pnpm qa
```

如果涉及图册来源或目录映射，重点看：

```bash
pnpm qa:catalogs
```

如果产品卡片影响移动端排版，再运行：

```bash
pnpm qa:browser
```

### 案例：新增酒店客房床头柜

正确做法：

1. 归类到酒店家具。
2. 准备名称、应用场景、图片、alt、可定制点。
3. 放入统一产品/图册结构。
4. 复用已有卡片样式。
5. 跑 QA。

错误做法：

1. 复制一个旧页面。
2. 改成床头柜。
3. 新写一个 `.bedside-card` 全局样式。
4. 以后每个产品都这样复制，网站很快失控。

## 6. 项目案例发布 SOP

### 适用场景

你要新增酒店、学校、办公室、医疗、住宅等项目案例时，使用这一章。

### 案例不是图片堆积

项目案例的作用是建立信任和引导询盘。它应该回答：

- 这是什么项目？
- 属于哪个行业？
- 使用了哪些空间？
- 我们供应了什么？
- 这个案例证明了什么能力？
- 访客看完以后应该联系谁或查看什么？

### 每个案例默认字段

- 项目名称。
- 行业分类。
- 项目地点或市场范围。
- 使用空间。
- 供应内容。
- 项目价值。
- 图片证据。
- 相关图册或产品入口。
- 联系 CTA。
- URL slug。
- SEO title 和 description。

### 操作步骤

1. 确认案例行业。
   - 酒店。
   - 办公。
   - 医疗。
   - 教育。
   - 住宅。

2. 查找现有项目数据。
   - 当前项目案例数据在：

```bash
src/data/site.ts
```

3. 复用项目列表和详情模板。
   - 不要为一个案例复制整页结构。
   - 不要因为案例图片不同就新写一套 CSS。

4. 准备图片证据。
   - 图片要能证明项目真实度或供应能力。
   - 图片 alt 要写清楚场景。

5. 加入 CTA。
   - 例如查看相关图册、发送项目参考图、联系项目顾问。

### 不要做什么

- 不要每个案例都新建完全不同的页面结构。
- 不要为了某个案例改全站 Header、Footer、按钮。
- 不要只放图片，不写采购价值。
- 不要复制相似文案堆案例。

### 检查方式

案例变更后运行：

```bash
pnpm qa
```

如果新增或调整了项目详情页，运行：

```bash
pnpm qa:browser
```

### 案例：新增一个酒店案例

正确做法：

1. 加到酒店项目分类。
2. 复用现有项目详情模板。
3. 配好项目名称、空间、供应内容、图片、CTA。
4. 跑 QA 和浏览器 QA。

错误做法：

1. 复制一整个项目页面。
2. 改文字和图片。
3. 临时写新 CSS。
4. 没有检查移动端。

### 哪些样式不能随便改

- `.btn`
- `h1`
- `.section-head`
- `.container`
- Header
- Footer
- 手机端菜单
- 弹窗
- 表单
- 全站颜色变量

这些样式通常影响很多页面。

### 操作步骤

1. 先判断要改哪个区域。
   - Header？
   - Footer？
   - 首页卡片？
   - 图册？
   - 手机端菜单？
   - 弹窗？

2. 找对应 partial。
   - 不要直接改 `global.css`。

3. 搜索选择器使用范围。

```bash
rg -n "selector-name" src
```

4. 小步修改。
   - 一次只改一个区域。
   - 不要同时改 Header、Footer、按钮、手机端菜单。

5. 跑检查。

```bash
pnpm qa
```

涉及页面视觉、Header、Footer、弹窗、手机菜单时，再跑：

```bash
pnpm qa:browser
```

### 不要做什么

- 不要继续往 `global.css` 写真实样式。
- 不要为了减少行数乱删 CSS。
- 不要看到重复 selector 就合并。
- 不要一边改 CSS 一边改 class 名。
- 不要把桌面端和手机端规则强行合并。

### 案例：想改 Header 高度

正确做法：

1. 先看 Header 基础样式。
2. 再看平板和手机响应式 partial。
3. 确认是否影响移动菜单。
4. 改完跑 `pnpm qa` 和 `pnpm qa:browser`。

错误做法：

1. 随便搜索 `.header`。
2. 改一个高度。
3. 不看手机菜单。
4. 不跑浏览器 QA。

## 8. CSS 审计结果怎么看

### 运行命令

```bash
pnpm qa:css:audit
```

这个命令不是自动清理工具。它只是告诉你 CSS 当前情况。

### 字段解释

- `cssFileCount`
  - CSS 文件数量。

- `cssLineCount`
  - CSS 总行数。

- `ruleCount`
  - CSS 规则数量。

- `duplicateSelectorCount`
  - 重复选择器数量。
  - 重复不一定是错误。

- `topDuplicateSelectors`
  - 最值得人工检查的一批重复选择器。

- `cssClassCount`
  - CSS class 总数。

- `sourceUnreferencedClassCount`
  - 源码里没有直接搜到的 class 数量。
  - 这个字段最重要。

- `importantCount`
  - `!important` 数量。

### 如何判断能不能删 CSS

优先看：

```text
sourceUnreferencedClassCount
```

如果是 `0`，说明没有明显可以安全删除的 class。不要为了减少行数硬删。

如果不是 `0`，也不能马上删除。因为有些 class 可能由脚本动态加上，或者只在弹窗、状态、移动端中使用。

### 为什么重复 selector 不一定是错误

例如：

```css
.nav-shell { ... }

@media (max-width: 720px) {
  .nav-shell { ... }
}
```

这表示桌面和手机分别有不同规则。这是正常的。

### 案例

如果审计显示：

```text
duplicateSelectorCount: 172
sourceUnreferencedClassCount: 0
```

正确理解：

- CSS 有重复覆盖，但目前没有明确废弃 class。
- 可以按区域慢慢整理。
- 不建议大规模删除。

错误理解：

- 看到 172 就以为有 172 个错误。
- 直接合并或删除。

## 9. 改完以后必须跑哪些检查

### 常用命令

```bash
pnpm qa:css
```

检查 CSS 拆分结构有没有被破坏。比如 `global.css` 是否还是入口清单。

```bash
pnpm qa:css:audit
```

生成 CSS 审计结果，帮助判断有没有可疑重复、未引用 class、`!important`。

```bash
pnpm qa
```

网站基础体检。包含 CSS 结构、构建、静态链接、图册数据等检查。

```bash
pnpm qa:browser
```

浏览器自动检查。会打开主要页面，看桌面端、手机端、菜单、弹窗、滚动卡片等是否正常。

```bash
pnpm qa:prelaunch
```

上线前检查。包含基础 QA 和上线配置检查。

### 什么情况跑什么

- 只改文案：至少跑 `pnpm qa`。
- 改图片路径：跑 `pnpm qa`。
- 改产品或图册数据：跑 `pnpm qa`，必要时看 `pnpm qa:catalogs`。
- 改 CSS：跑 `pnpm qa` 和 `pnpm qa:css:audit`。
- 改 Header、Footer、手机菜单、弹窗：跑 `pnpm qa` 和 `pnpm qa:browser`。
- 准备上线：跑 `pnpm qa:prelaunch`，必要时跑 `pnpm qa:browser`。

### QA 失败怎么办

1. 不要继续发布。
2. 看失败信息。
3. 回想最近改了哪些文件。
4. 只围绕失败点修复。
5. 修完重新跑同一个检查。

### 案例

你让 WorkBuddy 改了手机端菜单。它只说“已完成”，但没有跑 `pnpm qa:browser`。

处理方式：

要求它补跑 `pnpm qa:browser`，并报告移动菜单是否打开、是否溢出、联系按钮是否可见。

## 11. 常见问题排查 SOP

### 网站打不开

先检查：

1. 本地服务是否启动。
2. 端口是否正确。
3. 命令是否报错。
4. 最近是否改了配置文件。

不要先乱改页面。

### 页面样式乱了

先检查：

1. 最近是否改了 CSS。
2. 是否改了全站选择器。
3. 是否改了 Header、Footer、按钮、容器。
4. 是否只在手机端乱，还是桌面也乱。

运行：

```bash
pnpm qa
pnpm qa:browser
```

### 手机端横向滚动

先检查：

1. 最近是否改了卡片宽度。
2. 是否新增了固定宽度。
3. 是否改了移动端 CSS。
4. 是否改了图片或表格。

不要直接加 `overflow: hidden` 掩盖问题，除非确认这是正确兜底。

### 图片不显示

先检查：

1. 图片文件是否真的存在。
2. 路径大小写是否一致。
3. 文件名是否有特殊字符。
4. 页面里是否正确引用。

### Header 菜单异常

先检查：

1. Header 相关 CSS。
2. 手机端响应式 CSS。
3. 菜单按钮脚本或 data 属性。
4. 浏览器 QA 中 mobile menu 结果。

### QA 命令失败

先看失败在哪个阶段：

- CSS 结构失败。
- 构建失败。
- 静态链接失败。
- 图册数据失败。
- 浏览器 QA 失败。

不要看到失败就乱改多个地方。先修第一个明确错误。

### GitHub 检查失败

先看 GitHub Actions 输出。它通常会告诉你是构建失败、链接失败，还是脚本失败。

如果看不懂，把失败日志交给 WorkBuddy 或 Codex，让它先解释，不要直接改。

### 每次小更新后

至少运行：

- 图片是否存在。

- 内部链接是否合理。

运行：

涉及视觉时运行：

```bash
pnpm qa:browser
```

### 每月或每批运营后

建议做一次轻量巡检：

```bash
pnpm qa:css:audit
pnpm qa
```

看：

- `sourceUnreferencedClassCount` 是否明显增加。
- 是否有人把新样式写进了 `global.css`。

- 内容增长靠模板，不靠复制整页。
